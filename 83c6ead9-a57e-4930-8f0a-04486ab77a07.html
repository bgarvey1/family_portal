01 — Motivation

The problem it solves

Grandparents want to stay close to their grandkids' lives. But they're often left out of the informal, high-bandwidth family communication that happens over text threads, Instagram, and group chats. The information exists — it's just scattered, inaccessible to someone who isn't in those channels, and hard to query.

Meanwhile, kids and parents don't want to manage yet another communication platform. They want to keep using what they already use — their phones, Google Drive, iMessage.

The core insight: the problem isn't that grandparents lack access to family information — it's that no agent sits on top of that information and makes it conversationally accessible. A well-designed AI layer can bridge this gap without requiring anyone to change their behavior.

The secondary insight is that elderly users actually benefit more from a chat interface than younger users do. Natural language is more accessible than navigation, menus, and feeds. "What is Justin up to?" is a much lower barrier than scrolling through a photo grid.

02 — Product

What it does

👴 For grandparents
  • Large-text chat interface
  • Ask anything about the family
  • See shared photos inline
  • Live weather at the family's home
  • Weekly family podcast — press Play
  • No accounts, no navigation
👨‍👩‍👧 For family contributors
  • PIN-gated contributor portal
  • Add photos, notes, events, links
  • Drop media in Google Drive
  • AI classifies everything automatically
  • Admin can generate weekly podcast
  • Full vault management

The grandparent experience is deliberately minimal — one screen, giant text, a text box, and a send button. Everything else is handled by the AI or by family members behind the scenes.

The retrieval pattern (RAG)

The most important design decision is how the agent accesses the vault. Naively injecting all vault content into every prompt doesn't scale — after a few dozen items the context window fills and quality degrades. Instead the system uses a two-pass retrieval pattern:

1
Index read — Pass 1
Load the full manifest index from Firestore. Each manifest is a single compact line: ID, type, subjects, topics, description, mood, date. Even 500 items is only ~15KB — always fast, always fits in context.
GET /api/manifests → [ { id, subjects, topics, description, ... } ]
2
Relevance selection — Pass 1 continued
Send the question + full index to Claude. Ask it to return only the IDs of relevant items (max 5). This is a cheap, fast call — the model is routing, not generating.
Claude: ["a4f91b", "e1b88c"] ← selected IDs
3
Raw content fetch — Pass 2 setup
Fetch only the raw content for selected IDs. For photos, this returns a compressed JPEG thumbnail (≤200KB). For documents, the text excerpt. For links, the URL and metadata.
GET /api/raw/:id → { thumbnailDataUri, webViewLink, ... }
4
Grounded response — Pass 2
Build the final prompt: family context + retrieved vault items + photos as vision blocks. Claude generates a warm, specific, grounded response. Photos show inline below the message.
System prompt + retrieved items + [image blocks if photos] → response

The classifier

Every item that enters the vault gets classified exactly once — at upload time. For photos, Claude Vision analyzes the actual image. For documents, it reads the text. The output is a structured manifest:

// Example manifest — generated by Claude Vision from a photo
{
  "id":          "a4f91b",
  "type":        "photo",
  "title":       "Justin Hockey Tournament",
  "contributor": "Brendan",
  "date":        "2019-03-14",
  // ↑ lightweight fields — always loaded

  "subjects":    ["Justin"],
  "topics":      ["hockey", "sports", "tournament", "childhood"],
  "description": "Justin in full hockey gear celebrating after scoring
               a goal at Milton Academy rink, age 9, huge smile",
  "mood":        "joyful",
  "tags":        ["milestone", "childhood", "sports", "photo"],
  // ↑ classifier output — semantic search surface

  "rawRef": {
    "driveFileId":      "1BxiM...",
    "thumbnailDataUri": "data:image/jpeg;base64,..."
    // ↑ raw content — only fetched when selected
  }
}

The key architectural principle: the manifest is cheap to scan (a line of text), but rich enough for the model to make good retrieval decisions. The raw content is never touched unless the manifest is selected. This means the vault can grow to hundreds of items — years of family history — without any degradation in response speed or quality.

06 — Design Philosophy

A few principles worth preserving

Zero behavior change for contributors

The system is designed so that contributors — the kids, your wife — don't have to do anything differently. They use Google Drive like they might already. The backend does all the work of noticing, downloading, classifying, and indexing. The only ask is an occasional note when dropping in a video, because the AI can't watch video files yet.

The agent stays grounded

The system prompt is strict: answer only from the provided family context and vault items, never invent. This matters a lot for elderly users who might not know to question a confabulated answer. The source attribution pills (small amber tags under every AI response) show exactly which vault items the answer drew from — useful for building trust and debugging.

Scale without degradation

The index-first retrieval pattern means quality doesn't degrade as the vault grows. Adding a photo from 10 years ago doesn't make current responses worse. The classifier's rich descriptions are the mechanism — they have to be good enough that a future model can select the right item from a 500-item index based on a question it's never seen.

Local mode as a real fallback

The frontend runs fully in local storage mode with no backend at all. This was an intentional design choice — it makes iteration fast (no deploy cycle) and means the app can be demonstrated without any infrastructure. The BACKEND_URL and BACKEND_KEY constants at the top of the file are the only thing separating the two modes.

08 — What's needed

To get to a live demo

1
Install gcloud CLI + Node
Both available via Homebrew. One browser auth step, then everything runs from terminal. Claude Code can handle the rest of the GCP setup via CLI with no console required.
brew install --cask google-cloud-sdk && gcloud auth login
2
GCP project setup
Enable 5 APIs, create Firestore database, create service account, share the Drive folder. All scriptable — SETUP.md has the exact commands. ~15 minutes.
gcloud services enable run.googleapis.com firestore.googleapis.com ...
3
Deploy and wire up
gcloud builds submit + gcloud run deploy. Get the service URL. Paste it into the two constants at the top of FamilyHub.jsx. Done.
const BACKEND_URL = 'https://familyhub-backend-xxxx.a.run.app';
4
Populate and test
Fill in the family context in the Admin tab. Drop a few photos in the Drive folder. Hit Sync Now. Ask the grandparent chat "show me a recent photo." If it retrieves and displays the right image — it works.
